# Goalz.Infrastructure

## What is this?

This is the **data access layer** — it handles everything that involves talking to the outside world, primarily the database. It takes the abstract concepts from the Domain and Application layers and makes them concrete by actually reading from and writing to PostgreSQL (via Supabase in production).

---

## Why does this exist separately?

The rest of the application (Domain, Application) should never need to know *how* data is stored. By isolating database logic here, you can:
- Switch databases (e.g. from Supabase to a self-hosted PostgreSQL) by only changing this layer
- Swap out Entity Framework for a different tool without touching business logic
- Keep the rest of the codebase clean and focused

---

## What's inside?

### `Data/`
Contains the **DbContext** classes — these are EF Core's way of representing the database. Think of a DbContext as a live connection to the database that knows about all your tables.

| File | What it does |
|---|---|
| `GoalzDbContext.cs` | The main database context — maps all entities (User, Party, Quiz, etc.) to their database tables and defines the relationships between them |
| `AppDbContext.cs` | A secondary context used by the authentication system, focused on the Users table |

### `Repositories/`
Repositories implement the `IRepository<T>` interface defined in `Goalz.Domain`. They contain the actual database queries using EF Core.

| File | What it does |
|---|---|
| `Repository.cs` | A generic base repository that can get, add, update, and delete any entity from the database |

### `Migrations/`
These are **auto-generated files** created by EF Core when the database schema changes. You never edit these by hand — they are generated by running `dotnet ef migrations add <name>`.

Each migration file records exactly what changed in the database (e.g. "add a column", "rename a table") so the schema can be reproduced or rolled back reliably.

---

## How schema changes work

1. Edit an entity in `Goalz.Domain/Entities/`
2. Run: `dotnet ef migrations add <DescriptiveName> --context GoalzDbContext --startup-project ../Goalz.API`
3. Run: `dotnet ef database update --context GoalzDbContext --startup-project ../Goalz.API`

The migration is applied to whichever database your connection string points to (local Docker or Supabase).

---

## Dependencies

- **Goalz.Domain** — needs entities and interfaces to implement
- EF Core + Npgsql (NuGet packages) — the actual tools that talk to PostgreSQL

---

## Analogy

If this were a restaurant:
- **Infrastructure** = the kitchen — it does the actual work of preparing food (reading/writing data)
- It receives instructions from the waiter (Application) and uses the ingredients defined in the menu (Domain)
